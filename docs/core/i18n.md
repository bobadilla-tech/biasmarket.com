# Internationalization (i18n)

Bias Market ships with **Spanish and English from day one** — not retrofitted
later. Two things need translation, and they're handled very differently:

1. **Platform UI** (storefront chrome, dashboard, onboarding, emails, error
   messages) — written by Bias Market, ships in both locales.
2. **Seller-entered content** (store name, product names/descriptions, payment
   instructions) — written by the seller, in whatever language their store
   targets. Not translated by the platform.

Conflating these two is the main way i18n projects overbuild. Keep them
separate.

---

## Locale model

- `User.locale` — the seller's own UI language (dashboard, onboarding, emails to
  them). Set from browser `Accept-Language` at signup, overridable in account
  settings.
- `Store.locale` — the language the storefront renders in for _that store's_
  customers. Set once at onboarding (§3.1 in [product.md](product.md)),
  independent of the owner's own `User.locale` — a Spanish-speaking seller can
  run an English-language store.

MVP: **one locale per store.** A store's product listing, checkout copy, and
payment instructions render in a single language. No per-field translated
content, no locale switcher on the storefront. This mirrors the same reasoning
as deferring subdomains in
[architecture.md](architecture.md#3-multi-tenant-design-critical) — don't build
multi-locale content management until a store actually needs to sell to two
language markets simultaneously.

## Translation content package

```
packages/
  i18n/
    en/
      common.json
      landing.json
      dashboard.json
      storefront.json
      onboarding.json
      emails.json     (not yet built — no email feature exists)
    es/
      common.json
      landing.json
      dashboard.json
      storefront.json
      onboarding.json
      emails.json     (not yet built — no email feature exists)
```

- One shared package, imported by both `apps/web` (UI strings) and, later,
  `apps/api` (validation error messages, email templates) — strings live in one
  place, not duplicated per app. `apps/api` doesn't consume it yet; no backend
  localization has been built (see "NestJS: backend localization" below).
- ICU message format for plurals/interpolation
  (`"{count} {count, plural, one {product} other {products}}"`) — order counts,
  cart totals, etc. need this from the start, not bolted on later. `next-intl`
  (adopted below) uses ICU natively.
- Namespaced by feature (`landing`, `dashboard`, `storefront`, `onboarding`),
  not one giant `common.json` — keeps diffs small when a translator only touches
  one area. `common.json` holds only genuinely cross-cutting strings
  (loading/error fallbacks, page metadata).
- `packages/i18n/index.ts` exports a `getMessages(locale)` merging all
  namespaces into the shape `next-intl`'s `getRequestConfig` wants.

## Next.js: rendering strategy

**Decision reversed from the original MVP call below: this now uses URL-prefixed
locale routing (`app/[locale]/...`), not server-resolved no-prefix rendering.**
Use `next-intl` (App Router native).

- Every route lives under `app/[locale]/...` — `/es/...` and `/en/...`.
  `apps/web/proxy.ts` runs `next-intl`'s `createMiddleware` (Next.js renamed
  `middleware.ts` → `proxy.ts`), which negotiates the visitor's locale from
  `Accept-Language` on first visit and redirects to the matching prefix, then
  persists the choice in a cookie for subsequent visits.
- A manual switcher (`apps/web/components/landing/language-toggle.tsx`) lets
  visitors override the negotiated locale — implemented via `next-intl`'s `Link`
  with a `locale` prop, swapping the prefix while preserving the current path.
- `User.locale`/`Store.locale` remain in the schema as **stored preferences**
  (e.g. what locale a login redirect or a freshly generated store link defaults
  to), not as the runtime routing authority — the URL segment is the single
  source of truth for what actually renders, so there's no case where a DB field
  and the URL disagree about which locale to show.
- This applies uniformly to the landing page, dashboard, onboarding, and
  storefront — there's currently no split between "auth-gated, no prefix needed"
  and "public, needs a prefix" the original reasoning below assumed.

<details>
<summary>Original MVP reasoning (superseded)</summary>

- **Storefront**: locale comes from `Store.locale`, resolved server-side when
  the store is loaded (same place `TenantMiddleware`-equivalent resolves
  `store_id` on the frontend side) — no URL locale prefix needed, because a
  store is single-locale at MVP. Simpler than `/[locale]/store/[slug]`,
  consistent with the no-subdomain MVP scope.
- **Dashboard / onboarding**: locale comes from `User.locale` (session), with a
  manual switcher in account settings overriding it via a cookie. No URL prefix
  here either — these are auth-gated, not crawled, so SEO-driven locale URLs
  don't apply.
- Defer URL-prefixed locale routing (`/en/...`, `/es/...`) to v1, and only if a
  genuine need shows up (e.g. a store wants the _same_ storefront bilingual, or
  the marketing site needs locale-specific SEO pages).

## NestJS: backend localization

**Not built yet** — `apps/api` has no locale/i18n code at all today (confirmed
via grep across `apps/api/src`). Frontend UI copy is fully migrated to
`next-intl`; backend validation-message localization is a separate follow-up,
tracked here but out of scope until picked up.

- `nestjs-i18n` resolves locale per-request: `Accept-Language` header for
  public/storefront endpoints, `User.locale` claim from JWT for authenticated
  dashboard endpoints.
- **Validation error messages** (`class-validator`) — localize via the same
  package, not hardcoded English strings baked into DTOs. A Spanish-speaking
  seller filling out the onboarding form should see Spanish validation errors,
  not translated-in-the-frontend English ones.
- **Emails** (payment confirmation, order status changes — mostly post-MVP per
  [roadmap.md](roadmap.md)) — templated per locale from
  `packages/i18n/*/emails.json`, selected by the recipient's `User.locale` or
  `Store.locale` depending on whether the email goes to the seller or the
  customer. Sent via **Resend** (see
  [architecture.md](architecture.md#9-performance--scaling)) — the template
  resolves the locale, Resend just delivers the resulting HTML.

## Database implications

- `User.locale` and `Store.locale`: `String @default("es")` or `"en"` — no
  hardcoded platform-wide default; whichever the seller picks at
  signup/onboarding wins. Don't bias the schema toward one market.
- No locale-keyed JSON columns on `Product`/`Store` content fields at MVP —
  that's the v1 multi-locale-content feature, and adding it later is a additive
  migration (new nullable columns), not a breaking one. Don't pre-build it now.

---

## Explicitly out of scope for MVP

- Per-product/per-store multi-language content (translating the _same_ product
  into ES and EN simultaneously)
- Backend localization (`nestjs-i18n`, validation-message translation, email
  templates) — frontend UI is done, this is a separate follow-up
- Auto-translation (machine translation of seller content)
- Locales beyond ES/EN

## v1+

- If a store wants a bilingual storefront: locale-keyed JSON on
  `Product.name`/`description`/`Store.paymentInstructions`.
  (`/[locale]/
  store/[slug]` routing itself is already built — see "Next.js:
  rendering strategy" above — this is only about per-field translated
  _content_.)
- Additional locales as the market expands beyond ES/EN.
